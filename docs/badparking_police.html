Загальний огляд

BadParking (робоча назва) — технічний засіб передачі заяв про правопорушення з телефону заявника до поліції через сервер BadParking. Передача інформації до поліції відбувається через інтерфейс програмування сервісу BadParking (далі – API), використовуючи протокол HTTP(S) та формат JSON у схемі REST. Засіб використовує аутентифікацію клієнта API за допомогою пари ідентифікатор/секрет клієнта згідно опису в розділі „Аутентифікація клієнта“.

Даний документ розглядає аутентифікацію та лише ті точки взаємодії через API, які мають значення для подання заяви про правопорушення в поліцію та отримання зворотнього відгуку.

Загальна схема взаємодії має наступний вигляд:

Аутентифікація клієнта

Для початку аутентифікації необхідно отримати пару ідентифікатор/секрет. Пара генерується (або вводиться вручну через адміністративний інтерфейс BadParking) та передається клієнту заздалегідь обумовленим чином.

Процес аутентифікації використовується для всіх описаних даним документом точок взаємодії.

Наступні кроки описують алгоритм підпису REST запитів виданою парою:

  1. Отримати мітку поточного Unix-часу в секундах
  2. Сформувати підпис геш-функцією SHA256 від конкатенації секрету та результату з (1) таким чином: SHA256(client_secret + timestamp)
  3. Додати до параметрів GET запиту (або тіла POST/PUT запиту) ідентифікатор та підпис з (2) таким чином: client_id=<ідентифікатор>&client_secret=<підпис секрету>&timestamp=<мітка часу>

В разі вірної аутентифікації та відустності інших помилок сервер поверне відповідь із статус-кодом 200. В протилежному випадку буде повернено помилку з кодом 401.

Приклад реалізації алгоритму мовою Python:

 
xxxxxxxxxx
import time
from hashlib import sha256
​
client_id = '9a6291c1-ef6d-4d81-b6dd-a1699b4b78f0'
client_secret = 'testtest'
timestamp = str(int(time.time()))
secret_hash = sha256(client_secret.encode('utf8') + timestamp.encode('utf8')).hexdigest()
​
print("client_id={}&client_secret={}&timestamp={}".format(
    client_id, secret_hash, timestamp
))
# client_id=9a6291c1-ef6d-4d81-b6dd-a1699b4b78f0&client_secret=3c9348ebcd2e878c54ff94c606bfad255c3dbac2c58c3f6697cd786445aa49df&timestamp=1475789484

Інтерфейс подачі заяв

На данному етапі реалізовано метод подачі заявок через періодичні запити зі сторони поліції до спеціальної точки взаємодії з API. Дана точка реагує на запити типу GET та повертає дані списком словників з визначеною структурою. Кожен елемент списку є перевіреною на цілісніть заявою, яка ще не була оброблена поліцією на момент запиту (прим. — за даними сервісу, див. розділ „Пропозиції“).

HTTP адреса точки взаємодії: GET <домен API>/api/v1/claims

Можливі варіанти статусів заяви (поле status):

ЗначенняОпис
receivedОтримано сервером від заявника
completeЗаява пройшла автоматичну перевірку
enqueuedЗаява готова до обробки поліцією
acceptedОтримана поліцією
in_progressВ стані обробки поліцією
resolvedПоліція закрила заяву
invalidЗаяву відхилено (поліцією або модерацією)
canceledЗаяву відхилено заявником

Для поліцейського інтерфейсу доступні заяви з кодом enqueued. Коди accepted, in_progress, resolved та invalid має встановлювати інтерфейс зворотнього відгуку для відстеження ходу обробки заяви поліцією.

HTTP статус-коди відповіді:

  • 200 – запит виконано успішно
  • 400 – загальна помилка вводу
  • 401 – помилка аутентифікації
  • 500 – загальна помилка сервера

Інтерфейс отримання зворотнього відгуку

Для отримання зворотнього відгуку передбачено відправлення періодичних запитів до відповідного інтерфейсу на стороні поліції. Передбачається, що цей запит буде виду GET із схожою схемою аутентифікації та форматом обміну JSON. Також передбачається, що у відповідь інтерфейс поверне список заяв з їх відповідним ідентифікатором, статусом (одним з зазначених вище) та необов’язковим словесним описом у кожному елементі. У запиті буде передано список ідентифікаторів заяв, на які очікується відповідь.

Пропозиції

Інтерфейси, що описує даний документ побудовані на основі короткої переписки з Владиславом Криклієм як найбільш зручні для поліції на той момент. Проте можливі й інші схеми обміну інформацією, які описані в даному розділі.

Пропозиція №1. Подача заяв по одній в момент готовності з ініціацією стороною засобу BadParking напряму на поліцейський інтерфейс. Даний метод має суттєві переваги: дублювання реального процесу подачі заяви з формуванням „документа“, „підписом“ та однозначним підтвердженням отримання заяви поліцією; полегшення відстежування статуса конкретної заяви в будь-який момент часу; менше навантаження на системи за рахунок можливості оптимальнішої паралелізації.

Основний недолік поточної реалізації полягає в тому, що при заборі заяв поліцією неможливо отримати підтвердження того, що кожна конкретна заява дійсно отримана без подальшого запиту на інтерфейс зворотнього відгуку. Дана пропозиція усуне цей недолік.

Пропозиція №2. Зворотній відгук шляхом відправки запита до засобу BadParking зі сторони поліції в момент його надходження. Переваги такі самі як і в пропозиції №1: пришвидшення отримання відгуку заявником, зручніше відстежування статусу заяви, кращі можливості для паралелізації.
